# RFC-0007 Spike Results

Empirical answers to the open questions from
[`../0007-graphqlite-graph-layer.md`](../0007-graphqlite-graph-layer.md) §14.
Generated by running `test/spikes/graphqlite-spikes.test.ts` against
GraphQLite v0.4.4 installed via Homebrew on macOS arm64 (Darwin 25.3.0).

To reproduce:

```sh
brew install graphqlite
QKB_RUN_SPIKES=1 npx vitest run test/spikes/
```

## Q1 — Does GraphQLite `cypher()` participate in nested SAVEPOINTs? **YES**

Test: `BEGIN; cypher(CREATE outer); SAVEPOINT s1; cypher(CREATE inner); ROLLBACK TO s1; COMMIT`.

Result: only the `outer` node survives. SAVEPOINT semantics are preserved
across the SQL/Cypher boundary. The QKB SDK can rely on nested transaction
behavior for partial rollback patterns.

## Q2 — On-disk size of an empty graph

Measured by creating an empty SQLite DB with one trivial relational table,
loading GraphQLite, and forcing schema materialization with a single
no-op `CREATE/DELETE` Cypher cycle. File-size delta:

| Stage | Bytes |
|---|---|
| Empty SQLite + 1 marker table | 8,192 |
| After GraphQLite materialization | 196,608 |
| **Delta** | **188,416 (≈ 184 KB)** |

The RFC's original §10 budget of **64 KB is wrong**; the actual fixed
overhead of GraphQLite's schema (nodes table + edges table + indexes +
metadata) is roughly 3× that. Revising the threshold to **256 KB** in
the RFC, with this measurement as the rationale. This is a §1-class fix
to a hand-waved budget, not an architectural change — there is no smaller
empty-graph footprint to negotiate; this is what GraphQLite ships.

## Q4 — sqlite-vec and graphqlite coexist in the same connection? **YES**

Test: load both extensions on a single `Database` instance, create a
`vec0` virtual table, run an `INSERT` into the vec table and a Cypher
`CREATE` in the same transaction, COMMIT, then read back from each
extension.

Result: both round-trips succeed. Q4b (rollback) confirms that throwing
inside the transaction unwinds both extensions' writes atomically.

The vec0 virtual table requires a TEXT (or INTEGER) primary key — QKB's
existing schema uses `hash_seq TEXT PRIMARY KEY`. The spike test mirrors
this. An earlier version with implicit rowid hit
"Only integers are allows for primary key values" — fixed in commit by
matching QKB's actual schema.

## Q5 — Distribution: GraphQLite has no npm packages

Verified: no `graphqlite` / `@colliery-io/graphqlite` / per-platform
packages on npm. Distribution channels are GitHub releases + Homebrew +
PyPI + cargo.

Decision (confirmed by repo owner): **A+C hybrid vendoring**.

- Postinstall script (`scripts/postinstall-graphqlite.js`, to be added in
  PR-4) downloads from
  `github.com/colliery-io/graphqlite/releases/download/v{VERSION}/gqlite-{platform}`
  with SHA-256 verification. Lazy: runs only when needed (when the user
  enables the feature, not on every install).
- `QKB_GRAPHQLITE_PATH` env var lets airgapped/corp users pre-stage the
  binary and skip the download entirely.
- Pinned upstream version: **v0.4.4** initially. Review at each upstream
  release; track in `scripts/graphqlite-versions.json`.
- Platform matrix: Linux x86_64, Linux arm64, macOS arm64, macOS x86_64.
  Windows deferred (upstream marks Windows builds `continue-on-error` in
  v0.4.4 due to a runner regression).

## Q3 — Dump format: GraphQLite-native or QKB-defined? **STILL OPEN**

This is a taste call escalated to the repo owner via PR-2. Tradeoffs:

| Option | Pro | Con |
|---|---|---|
| GraphQLite-native (Cypher script) | One fewer thing to maintain; round-trips natively if upstream adds an import tool | Coupled to GraphQLite forever; format changes shipped from upstream propagate immediately; harder to port to a different graph backend |
| QKB-defined NDJSON (one node per line, then one edge per line, with `kind` discriminator) | Decouples us from GraphQLite; readable; easy to migrate to Kùzu or any other backend; matches the §7 "exit plan" framing | Slightly more code; we own format-evolution risk |

Recommendation: **QKB-defined NDJSON**. It is the format that makes the
exit-plan claim in §7 actually mean something — if we used the
GraphQLite-native format, our exit-plan tool just hands the user the
exact same problem they're escaping. The implementation cost is
~50 LOC.

## Cypher API signature correction

Research (PR-2 follow-up to upstream docs) confirmed the actual SQL signature:

```sql
cypher(query TEXT [, params_json TEXT]) -> TEXT  -- JSON-array of result objects
```

— **two args, not three**. The original RFC text suggested
`cypher('namespace', query, params)`, which is wrong.

Implications applied to the RFC:

1. §4.2 "namespace" framing replaced with single-graph-per-DB. The
   `qkb` namespace literal is removed. Multi-graph in GraphQLite is via
   Cypher's `FROM <graphname>` clause (read-only, max ~10), not a
   function-arg namespace.
2. §4.6.1 SDK template-tag pattern is unchanged (it was already a
   2-arg `(query, params)` shape at the SDK level).
3. §9 schema-bootstrap step removes "registers the `qkb` namespace" —
   nothing to register. Just ensure `graph_meta` table.
