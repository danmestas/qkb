Indexed Caveman Bench — does compressing the *indexed* hot.md change retrieval?

2026-05-15 · 3 levels × 2 questions = 6 router runs · cave hot.md indexed into a separate flight-planner-cave qkb collection on a git worktree · isolated from production index

Setup

Earlier today's unindexed-cave bench tested only the priming-context dimension — the cave files lived in /tmp/ outside the qkb collection, so the agent's qkb search calls hit the unchanged production index. That left an open question: does putting the cave content in the index change search rankings? Maybe a denser BM25 chunk pulls hot.md above the canonical entity/concept pages on questions where hot.md mentions the same terms.

To test this without disturbing the production setup:

  • git worktree add --detach .worktrees/cave-index-bench main on flight-planner-kb — isolated working tree at main's tip (committed state only; main's uncommitted edits don't leak in)
  • qkb collection add <worktree-path> --name flight-planner-cave — separate index, doesn't touch production flight-planner collection
  • For each level [std, high, ultra]: cp /tmp/qkb-cave-bench/hot.cave.{level}.md → worktree/wiki/hot.md, then qkb update, then dispatch FP-1 + FP-2 (parallel) using the unmodified production vault-query skill with -c flight-planner-cave override

Findings

  1. Indexing the cave hot.md is a no-op for retrieval. Token counts match the unindexed bench within ±5% (i.e., within run-to-run noise of the subagent harness). Quality unchanged (30/30 across the board). Whether the cave content sits inside the qkb index or outside it, the agent answers the same way with the same cost. This was the core hypothesis the bench tested, and the answer is clean.
  2. Why: hot.md isn't a search target. On FP-1, every condition's top BM25 hit is concepts/Briefing-Assembler.md at 0.84. On FP-2, top hits are pattern concept pages (Cycle-Aware-Publication-Refresh, Schema-Drift-Manifest). hot.md is never the top hit and never breaks into the top 3 on these questions. It's a hub file — read by the agent as priming context via direct Read tool calls, not surfaced as a search result. So compressing it leaves BM25's preferred retrieval targets unchanged.
  3. One small anomaly worth noting. FP-2 cave-high shifted its top BM25 hit from Cycle-Aware-Publication-Refresh.md (the std/ultra top hit) to Schema-Drift-Manifest.md. Both are valid pattern pages; both got cited in the answer. Possible explanation: at 39 KB, cave-high's hot.md chunked into fewer chunks with denser term concentrations, slightly shifting BM25's idf computation across the collection. Effect was real but did not change answer quality or token cost meaningfully.
  4. cave-ultra was again the cheapest. FP-1 ran in 40.7s (vs 49.5s for high, 65.4s for std) and 51,250 tokens — the lowest of the indexed conditions. The non-monotonic cost curve from the unindexed bench reproduces: aggressive compression (cave-high) triggers extra retrieval; moderate compression (cave-ultra ~55% size) is the sweet spot.
  5. Answer quality held at all indexed levels, same as unindexed. All six indexed runs scored 30/30. Cave conditions surfaced more architectural patterns on FP-2 than the prior bench's prose baseline — the same effect previously attributed to "bullet-form is more enumerable" appears again. Indexing the cave content doesn't enhance this; it's a function of the priming context shape, not the search corpus.

Implication for the original "shrink the always-read" thesis

Two benches now triangulate the same conclusion: hot.md compression is the wrong intervention for this vault.

  • Unindexed bench (priming-context only): compression saved at most 4.9% tokens at the sweet spot (cave-ultra), and at aggressive levels (cave-high) cost 31% more tokens via extra retrieval.
  • Indexed bench (priming-context + search corpus): token spend matched the unindexed bench within ±5%. Indexing added no benefit, because hot.md isn't where the BM25 searches go. The agent's qkb search calls target entity/concept pages regardless of what's in hot.md.

The two interventions together — compress hot.md's bytes (read-time) and compress hot.md's index chunks (search-time) — cumulatively do not move the bench's outcome. The cost story belongs elsewhere: page reads, harness overhead, or synthesis budget. Token spend is a fact about where the agent looks, not what the hub file says.

The next interventions worth testing are the ones that target what BM25 actually surfaces:

  • BM25-chunk index.md (24k tokens of "Page Name: one-line desc" rows) — replace "read whole" with "search for query, return top-K rows." Different artifact from hot.md: index.md is structured as a lookup. Best remaining shot at the always-read tax.
  • Shorter / denser entity pages — the actual top BM25 hits. If concepts/Briefing-Assembler.md is 140 lines and the answer needs only the frontmatter + first paragraph, "frontmatter-as-answer" page rewrites could cut Read costs at the part of the budget that does dominate.
  • Profile per-tool token costs — every run did ~5–10 tool calls each costing different amounts. The bench captured totals but not breakdowns. A profiling pass would show whether page reads, qkb search outputs, or harness overhead dominate the 50–90k-token budget.

Methodology

Reversibility design: a git worktree on flight-planner-kb (detached HEAD at main tip) isolated the experiment from the production checkout. A separate flight-planner-cave qkb collection was created against the worktree, so the production flight-planner collection (which other tools depend on) was never modified. For each level, the worktree's wiki/hot.md was overwritten with the cave version and qkb update re-indexed only that file (because qkb tracks file hashes and other files were unchanged).

6 retrieval runs dispatched as parallel general-purpose subagents (2 per level, fan-out=2). Each subagent received the unmodified production vault-query SKILL.md (same skill the prior bench used) with an instruction to override the collection name from flight-planner to flight-planner-cave. Read paths pointed at the worktree.

Caveat: the flight-planner-cave collection has 808 indexed files vs the production flight-planner collection's 1370 (the production vault has 562 untracked site/content/ files from a Quartz static-site build that aren't in the committed worktree). The bench questions concern wiki/ pages, which are identical in both collections — but the BM25 idf statistics differ slightly because the corpus size differs. This is a known confound; it likely accounts for some of the ±5% noise vs the unindexed bench. The conclusion (indexed-vs-unindexed makes no meaningful difference) is robust to this confound because the effect size we'd be measuring is much larger than the BM25-statistics shift.

Quality scoring used the same v5 rubric (coverage / accuracy / synthesis 0–5). All six indexed runs scored 5/5/5 = 15/15 per question = 30/30 total per condition.
