Setup

The first three benches couldn't decide whether qkb was worth maintaining — vault-query (no engine) was competitive with the qkb-backed skills on benign questions. v4 was designed to break that ambiguity. Six adversarial questions, each engineered to expose a specific failure mode in the cohort. Pre-committed decision rule: if qkb's best skill beats vault-query by ≥ 15 points → keep qkb. If < 5 points → drop qkb.

Decision rule application

The pre-committed rule:

  • qkb's best beats vault-query by ≥ 15 pts → keep qkb (clear) — DID NOT TRIGGER. qkb's best (bm25/graph at 78%) is below vault-query (85%).
  • Spread < 5 pts → drop qkb (clear) — DID NOT TRIGGER. 7-point spread.
  • 5–15 pt gap → ambiguous, need another round — applies, BUT the gap is in the wrong direction (vault-query ahead).

Naïve reading: "vault-query beat qkb, drop qkb." Adjusted reading: vault-query only "won" by silently using qkb on 3 of 6 questions. Strip the qkb fallback and vault-query loses Q-A, Q-C, and Q-E — three of six. Pure-curated vault-query would land around 60% aggregate, not 85%.

The real finding

No single path dominates the adversarial set. Five paths, five-way coverage between 71% and 85% (with the 85% asterisked). Per-question winners are spread across 4 of the 5 paths. The bench doesn't converge on one algorithm — it converges on the fact that different question shapes need different retrieval primitives.

Three loadbearing observations:

  1. vault-query is a router, not a retrieval engine. The agent reads hot.md/index.md, decides what to do next, and reaches for qkb commands when the curated path fails. In practice, this is the cohort's most successful pattern — combining the cheap default (curated read) with selective qkb escalation.
  2. The hybrid pipeline (qkb query --graph) doesn't justify its latency. 73% coverage, 80s wallclock, came 4th overall. Won 2 questions by tying, dominated none. The 30-60s of LLM expansion + reranker compute doesn't translate into uniquely-good answers.
  3. PPR is narrow. 71% aggregate but with a 20% catastrophic failure on Q-F (hub-bias confirmed). PPR works on Q-B / Q-C shapes (pairwise / pattern-set with named entities) but doesn't generalize. The RFC-0010 v2 (degree-normalized PPR, issue #89) is needed before PPR is a viable general-purpose path.

Convergence path: 5 skills → 3 skills + a router

Based on this bench, the cohort can consolidate without losing coverage. Drop the bottom two paths, keep the three that own distinct question shapes, and document the routing logic.

Keep:

vault-query-graph Q-E (cold page) The only path that uniquely beats the field on cold pages and relational walks. Pure-Cypher discipline (no BM25 fallback in retrieval) is justified.

What gets implemented for the router

Currently the vault-query agent has implicit routing: when curated reads fail, it pragmatically shells out to qkb. Make that explicit:

  1. Read hot.md. If question is current-context-heavy and answers there: done.
  2. Read index.md if it fits. If question maps to a named page in the index: read it, done.
  3. Else: classify question shape.
    • Proper-noun anchor or distinctive vocab → run qkb search (bm25 path).
    • Relational / entity-relation / cold-page → run qkb graph Cypher (graph path).
    • Ambiguous → run qkb search first, escalate if top-5 are weak.
  4. Always batch-read survivors via qkb multi-get (saves tokens).

This is what the agents are already doing implicitly. Making it explicit cleans up the cohort, eliminates the "vault-query secretly uses qkb" confusion, and gives the cohort a clear story: one default skill (the router), two specialist primitives (bm25, graph) it dispatches to.

The actual question: is qkb worth maintaining?

Yes, but narrower than I'd previously framed. The case for keeping qkb:

  1. vault-query secretly depends on it. Strip qkb and vault-query loses 3 of 6 adversarial questions. The curated-read path needs an escape valve.
  2. BM25 is doing real work. 78% aggregate, fastest path (~30s), wins or ties on Q-A, Q-C, Q-D. The FTS5 index + collection scoping is load-bearing infrastructure.
  3. Graph layer is uniquely necessary for cold pages. Q-E (Route Server, not in hot.md / index.md) — graph wins at 92%, BM25 lands at 58%, vault-query needs qkb help. Without the graph layer, cold-page questions become unanswerable as the vault grows.
  4. Multi-vault scoping (the -c flight-planner / personal pattern) is qkb-native. vault-query can't span collections.

The case for trimming qkb:

  1. The hybrid pipeline isn't earning its keep. Vector + LLM expansion + rerank cost ~50s of wallclock and dropped the skill to 4th place. Could be removed, freeing up significant maintenance surface.
  2. PPR is on hold. The RFC stays, the spike stays, but Phase 1 implementation doesn't ship until issue #89 (degree-normalized PPR) is benched. Don't add code that 71% bench coverage justifies.

Recommended scope for qkb:

  • Keep: FTS5 BM25 index, sqlite-vec embeddings (used by hybrid which is now optional), graph layer + Cypher engine, qkb search, qkb graph, qkb get / multi-get, collection scoping, qkb update.
  • Demote (still ship, but not the default): qkb query --graph hybrid pipeline. Keep for users who want maximum recall and don't mind 60s wallclock, but don't push it as the primary path.
  • Hold: qkb graph rank (RFC-0010 Phase 1). Reopen if issue #89 ships a viable v2.

This shrinks the surface qkb has to defend without losing the load-bearing capabilities the bench validated.

Next steps

  1. Reframe vault-query as the router skill. Update its SKILL.md to explicitly document the curated-read-then-escalate pattern instead of leaving it as implicit pragmatism. Three-step flow: hot.md, index.md, then dispatch to bm25 or graph based on question shape.
  2. Drop vault-query-qkb (the hybrid pipeline skill). 4th place on the adversarial bench, slowest, no unique wins. The qkb query --graph command stays in the qkb CLI (advanced users can call it directly) but it's not a default skill in the cohort.
  3. Hold PPR. RFC-0010 stays merged for the record. Phase 1 implementation doesn't ship until issue #89 (degree-normalized PPR) has a working spike that fixes the hub-bias.
  4. Investigate the discriminator on Q-E. Three skills (bm25, ppr, hybrid) all called CWDS a "weather overlay" — actual canonical is "Combat Weapons Delivery System." vault-query and graph got it right. This is a meaningful correctness failure that lexical retrieval (BM25) and statistical retrieval (vector) both made, but graph traversal avoided. Worth understanding the failure mode.
  5. Re-bench after the consolidation — 3 skills × 6 questions, fresh agents, measure the no-leakage version of vault-query routing. Decision: does the explicit router maintain the 85% level, or does forcing the routing explicit cost coverage?
