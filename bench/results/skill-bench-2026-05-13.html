TL;DR

  • The first bench was wrong about BM25. Rigorous fact-coverage scoring shows bm25 trails the other three by ~16 points and has accuracy failures (Q6 hallucination, Q4 broken wikilink, Q1 misframed update mechanism).
  • vault-query and qkb are statistical ties on the questions where vault-query can actually do its job. But vault-query degraded badly on Q5 (20% coverage) and Q4 (64%) — questions where the answer is scattered across many pages.
  • vault-query-graph is reliable but prone to scope creep. Best on Q6 (freshness comparison), but on Q4 it dragged in EUROCONTROL Digital NOTAM coverage that didn't answer the U.S.-NMS-loss question, and inverted Notamify's NMS-dependency from a risk into a strength.
  • vault-query-qkb is the most balanced. Never sub-71% except on Q5, never hallucinates, citations always resolve. It earns its 2× latency on hard questions but has no edge on easy ones.

Methodology

  1. Question selection — 6 questions covering 4 shapes: proper-noun anchor (Q1), distinctive-vocabulary (Q2), vocabulary mismatch (Q3, Q5), multi-hop comparison (Q4, Q6).
  2. Execution — Each question dispatched to 4 subagents in parallel, one per skill. Each agent read its SKILL.md, followed the documented flow end-to-end, timed itself between first retrieval action and final synthesis. No vault edits, no qkb update/embed.
  3. Ground truth — After all 24 answers were collected, a separate evaluation agent per question read the canonical 2–4 vault pages and extracted 10–15 specific, verifiable facts that a correct, complete answer should mention.
  4. Scoring — Each answer was scored against the fact list (paraphrasing accepted) and its wikilink citations were verified against the actual file tree (find wiki/ -iname '...').
  5. Caveats — Per-question batches ran in parallel, so the two qkb query --graph agents contended on the reranker model (inflating wallclock for Q2/Q5 by ~2×). Quality scoring is at fact-coverage granularity; one fact-list item is one binary checkbox, so subjectivity in fact-list construction is the main source of noise.

Ground truth (15 facts)
  1. NMS is FAA's cloud-based NOTAM Management Service; cut over April 18, 2026, replacing legacy USNS/FNS.
  2. Built/operated by CGI Federal on GCP; base https://api-nms.aim.faa.gov/nmsapi.
  3. REST API with OpenAPI 3.0.4 spec v1.0.17; machine-to-machine only (browser → 401).
  4. Auth: OAuth2 client-credentials via POST /v1/auth/token; bearer-token TTL ~30 min; no /nmsapi/ prefix on auth.
  5. Two-phase ingestion: Initial Load on each new 28-day AIRAC cycle, then continuous Delta Polling.
  6. Initial Load via GET /v1/notams/il — full corpus as AIXM 5.1 in SOAP envelope.
  7. Large payloads via GCP signed URLs proxied through /v1/content/{token}; URLs expire after 5 minutes; bearer required.
  8. Delta Polling via GET /v1/notams filtered by lastUpdatedDate, hard 24-hour max lookback.
  9. Response format selected per-request by nmsResponseFormat header (AIXM | GEOJSON); GeoJSON preferred for deltas.
  10. Five classifications: DOMESTIC, INTERNATIONAL, MILITARY, LOCAL_MILITARY, FDC.
  11. Twenty filterable features; /v1/notams takes 12 query params, requires ≥1 filter.
  12. Geospatial filter: lat+lon+radius together; radius ≤100 nm.
  13. /v1/locationseries provides location-code mapping with ≤5-day lookback.
  14. NMS will publish NMS-NPS feed back into FAA SWIM as a follow-on.
  15. Verified at scale 2026-05-02: 74,672 NOTAMs across 4 classifications.

Ground truth (15 facts)
  1. Schema-Drift Manifest pattern: pins expected upstream structure, fails fast on mismatch.
  2. Pattern originates from PreflightAPI's defense against aviationweather.gov changes.
  3. Pain is asymmetric: upstream changes publish in minutes, downstream discovery takes days.
  4. Implementation A: PreflightAPI per-source JSON manifests for METAR/TAF/PIREP/SIGMET/G-AIRMET.
  5. Implementation B: Firestorm shared source_schemas table (migration 0005), reused across NMS-API, NASR, OurAirports, AWC.
  6. Fingerprint = sha256 over sorted union of top-level JSON keys.
  7. Row schema: (source, file, fingerprint, first_seen_at, last_seen_at, blessed); blessed: 0=unseen, 1=accepted, 2=broken.
  8. Ingest proceeds regardless of blessed state; emits via IngestSummary for observability.
  9. Fingerprint catches key-set drift only; type drift (visibility_sm REAL → "6+") needs decoder validation.
  10. Atomic Pipeline Handler emits SchemaBlessed per upsert.
  11. Lineage Invariant: every parsed row carries source_object_key NOT NULL → R2 archive.
  12. AWC tables also persist raw_text and raw_payload (NOT NULL) for in-place re-parse.
  13. Lineage enables schema-drift diagnosis: byte-level diff of original R2 bytes; reparse with bumped parser_version.
  14. Transient sources (NOTAM, Wx) have stricter freshness SLOs; stale transient = operationally unsafe.
  15. Schema-Drift Manifest + Lineage Invariant cover both static and transient sources.

Ground truth (15 facts)

SAME (5 facts):

  1. Both use FAA/regulatory grounding (14 CFR §91.103 for NOTAM, FAA AIM 7-1-26 for weather).
  2. Both share the same three time-states: active_in_window / upcoming_in_window / background.
  3. Both YAML-tunable / operator-configurable.
  4. Both surface staleness explicitly rather than silently dropping.
  5. Both extend to other safety-critical feeds (NOTAM page says weather/SIGMET/TFR each gets its own freshness oracle).

DIFFERENT (10 facts):

  1. Enforcement layer: NOTAM gate at firestorm-api boundary; weather rules in Briefing Assembler at render time.
  2. NOTAM single 60-min SLO vs weather 3-bucket gradient (fresh ≤55 / aging 55–110 / stale >110).
  3. Threshold rationale: NOTAM 60-min is "looser end of defensible band"; weather buckets are 2× and 4× 30-min METAR cadence.
  4. NOTAM stale → HTTP 503 + Retry-After: 60; weather stale → red timestamp + "STALE" marker + medium-severity OW flag.
  5. NOTAM refuses to serve; weather still ships brief annotated.
  6. NOTAM is per-classification (FDC stale doesn't block DOMESTIC); weather is per-station/per-product.
  7. NOTAM trips fire Sentry tagged data_loss_risk: true; weather goes to Operational Warnings.
  8. NOTAM 200 carries valid_as_of timestamp.
  9. NOTAM gate wired (PR #112, ADR-0010); weather weatherWarnings stubbed (Fixture 6 documents gap).
  10. Weather has aging middle band (55–110, italic); NOTAMs have no middle band — just fresh/stale.

Action items for the skill cohort

  • BM25 needs a hallucination guardrail. When top-K hits come back with weak scores or non-wiki paths (specs/, archive/), the skill should refuse synthesis and escalate to qkb rather than ground in stale content. The Q6 failure mode is the safety case.
  • Drop the "AB test" framing on vault-query-graph. It's a working production retrieval skill — not an experiment. Keep both qkb and graph since they specialize: qkb for synthesis, graph for pairwise comparison.
  • Skill descriptions should bake in triage hints. The "when to use" advice above is currently invisible to the skill-selection layer; move it into each skill's description frontmatter.
  • URI emission: hyphenated vs space-preserving. 3 of 4 graph/qkb agents hit URI-to-path friction (qkb emits Schema-Drift-Manifest.md but actual file is Schema-Drift Manifest.md). Either preserve spaces or document the conversion in the skills.
  • vault-query needs a scale-aware fallback path. On Q4 and Q5 it ran for 200+ seconds and still degraded to 20-64% coverage because hot.md/index.md exceed read budget. Add an explicit "delegate to qkb if meta-files don't fit" branch.
