Slice-Fetch Bench — does qkb get :line -l 40 beat full Read?

2026-05-15 · 2 conditions × 2 questions = 4 router runs · baseline (production vault-query skill) vs slice-fetch variant (explicit qkb get with line targeting, no Read tool on entity pages)

Hypothesis

Speculation from the post-bench discussion: agents reading whole entity/concept pages via the Read tool spend most of the per-query budget on file content the question doesn't need. Replace "Read N pages" with "qkb get <path>:<line> -l 40" — a 40-line slice centered on the BM25 hit line — and recover ~50% of input tokens.

The slice variant modified the production skill in three places: standard-mode step 5 ("Read 3–5 candidates" → "Slice-fetch 3–5 candidates via qkb get :line -l 40"), the index-pointer step (same swap), and deep-mode batch-fetch (multi-get with -l 60).

Headline

Hypothesis failed. Slice-fetch as implemented made things slightly worse, not better. FP-1 spent 16% more tokens and 42% more wall-clock; FP-2 was effectively flat (+2% tokens, +4% latency). Quality unchanged at 15/15 both ways. Sign-off declined.

Why the intervention didn't help

  1. The "Read whole files" failure mode I was targeting wasn't actually happening at baseline. Look at the baseline column's read_tool_calls: 1 and qkb_get_calls: 1–5. The baseline agents were already using qkb multi-get / qkb get organically, because the production skill mentions multi-get explicitly in the deep-mode section. The agents read instructions holistically and didn't actually default to full Read on entity pages. The cost I thought I was attacking was a phantom.
  2. Forcing per-file qkb get :line traded fewer-but-bigger tool calls for more-but-smaller ones — and lost. Baseline does 1 batched qkb multi-get call returning N pages. Slice does N separate qkb get :line calls. Each tool call carries harness overhead (system prompt injection, response wrapping). On FP-1 that overhead exceeded the content savings: 4 slice calls cost more total tokens than 1 multi-get even though each individual slice returned less text.
  3. The patched skill is itself bigger (12.1 → 13.2 KB). ~270 extra tokens just to load the slice-variant skill. Small but contributes to the regression.
  4. Run-to-run variance is significant. Today's baseline ran cheaper than the prior bench's baseline by 11–14% (FP-1: 53,293 → 45,938; FP-2: 67,144 → 59,556). Same skill, same vault, same model — different runs. The +2% to +16% slice-vs-baseline gap in this bench is within that variance window. The gap could be noise.

What we learned

  • The production skill is already efficient on this vault. Three benches now (caveman unindexed, caveman indexed, slice-fetch) have failed to find an intervention that reliably beats it. Token spend lives somewhere the priming/page-fetch layer can't reach.
  • Batched multi-get beats per-file slice on this question shape. Tool-call overhead matters at the 10+ k-token scale. Future skill changes should optimize for fewer tool calls, not smaller tool calls.
  • Bench reliability needs N>2. Two-question benches can't distinguish a +5% intervention from a +5% noise floor. To resolve a "small but real" win like a slice-fetch tweak might be, the bench needs ≥6 questions per condition. The cheap synthesis questions (FP-2 shape) are particularly noisy.
  • The next experiment should profile per-tool token costs, not assume where the slack is. Three failed-or-flat benches in a row is the system telling us we don't know what we're attacking. A profiling pass on a single v5-style run — capturing tokens per tool call — would settle whether the dominant cost is hot.md reads, entity page reads, qkb multi-get outputs, or harness overhead.

Reversibility

Nothing in flight-planner-kb modified. Skill variants lived only in /tmp/qkb-slice-bench/ throughout. Production vault-query skill untouched. Production flight-planner qkb collection untouched. Single rm -rf /tmp/qkb-slice-bench/ restores pre-bench state.

Methodology

4 retrieval runs dispatched as parallel general-purpose subagents. Each received a path to either skill.prose.md (a copy of the production skill) or skill.slice.md (the same skill with sections 1, 5, and deep-mode batch-fetch rewritten to use qkb get :line -l 40 / qkb multi-get -l 60 instead of Read). Both pointed at the production flight-planner qkb collection. Subagents were instructed to operate read-only.

Token / latency captured from each subagent's <usage> block. Tool-call counts self-reported in the bench metadata block. Quality hand-graded on the v5 rubric (coverage / accuracy / synthesis 0–5).
