What changed since v1

The v1 bench surfaced two architectural problems in the skill cohort:

  1. vault-query-graph was a functional duplicate of vault-query-qkb — both ran qkb query --graph. Same retrieval engine, different prose. PR #40 rewrote it as pure graph traversal using qkb graph neighbors and qkb graph query Cypher — no BM25, no vector, no LLM expansion, no rerank.
  2. vault-query-qkb wasn't using qkb's full toolkit — it was a single-shot wrapper around qkb query --graph. PR #40 rewrote it to orchestrate the breadth: qkb multi-get for batched content retrieval (one call vs 5 Read invocations), qkb graph neighbors as one-hop enrichment on top hits, multi-pass query strategy in deep mode, parameter-tuned weak-hit recovery (--graph-weights, --no-rerank, --no-graph).

The v1 bench also exposed real schema bugs in my earlier graph skill docs: graph nodes use integer-form doc:<N> IDs (not the FTS #<hash>), label is :T (not :Note), LINKS_TO is typically the only edge type populated in Obsidian vaults. All examples corrected.

TL;DR

  • vault-query-graph v0.2.0 is now the top skill across all 3 questions. The pure-graph-traversal rewrite + correct schema (integer doc:<N> IDs, Cypher-based seed lookup) gives it a clear advantage on relational and pairwise questions. Latency dropped from ~100s+ in v1 to ~52s in v2 because there's no reranker contention.
  • vault-query-qkb v0.2.0 improved modestly (+5pp). The full-toolkit orchestration helped — Q9 used multi-get to batch-pull 4 files in one call, Q10 used multi-get + graph enrichment. But the answers didn't dramatically improve; the gains were marginal relative to the latency cost.
  • vault-query-bm25 at 83% is the value play. 2× faster than graph, only 5pp behind on coverage. For high-volume / latency-sensitive use cases, BM25 + careful skill discipline beats the full pipeline.
  • vault-query at 76% is competitive but not differentiated. Q9 burned 170s because hot.md exceeded read budget and it had to fall back to qkb commands. When meta-files fit, it works fine; when they don't, it degrades.

The v1 bench's biggest claim was "BM25 wins on speed AND quality." v2 invalidates the quality half — graph v0.2.0 beats BM25 on coverage by 5pp, and the gap widens on relational questions (Q10: graph 96% vs bm25 92%; Q11: graph 82% vs bm25 71%). The retrieval-vs-cost frontier is more nuanced than v1 suggested.

Ground truth (17 facts)
  1. Schema-Drift Manifest pins expected upstream structure, fails fast on mismatch.
  2. Symptoms: dropped new fields, renamed → NULL, removed → break far-downstream, type narrowing.
  3. Asymmetry: upstream publishes in minutes, downstream discovery takes days.
  4. PreflightAPI (Impl A) embeds per-source JSON manifests, validates before parse.
  5. Firestorm Dataworks (Impl B) uses shared source_schemas table (migration 0005).
  6. Fingerprint = sha256 over sorted union of top-level JSON keys.
  7. Upsert columns: (source, file, fingerprint, first_seen_at, last_seen_at, blessed).
  8. Blessed values: 0 unseen, 1 accepted, 2 broken.
  9. Ingest worker proceeds regardless; emits state in IngestSummary.
  10. Tradeoff: fingerprint catches only key-set drift; type drift in decoder.
  11. AWC quirks: seriesId/tag, stringified-float coords, epoch vs ISO times.
  12. Setup ~1 day/source; saves ~2 days per incident.
  13. Pre-condition for Two-Tier Storage parsed-tier ingestion.
  14. Complements Lineage Invariant — defines "correct shape" for fetches.parsed JSONB.
  15. Atomic Pipeline Handler emits SchemaBlessed in IngestSummary.
  16. Applies to: NOAA AWC, FAA NASR, FAA DOF, FAA NMS ArcGIS airspace layers.
  17. Lineage Invariant: every parsed row carries source_object_key → immutable R2; AWC rows also persist raw_text/raw_payload.

Findings and recommendations

The graph skill's transformation is the headline

v0.1 (hybrid path with AB framing): 72% avg, frequent scope creep, ran the same command as qkb.
v0.2 (pure graph traversal, fixed schema): 88% avg, top of every question in this bench, half the latency.

The lesson: skill specialization matters more than skill quality. v0.1 was a slightly-different way to call the same command; v0.2 is a genuinely different retrieval shape that wins on the questions matching its discipline.

qkb v0.2's full-toolkit orchestration helped, but less than expected

v0.1 (single-shot qkb query --graph): 74% avg.
v0.2 (multi-get for batch reads + optional graph neighbors enrichment): 79% avg.

Modest improvement. The agents used the new commands as designed (Q9: multi-get pulled 4 files in one call; Q10: hybrid + multi-get), but the answer quality didn't dramatically improve. Hypothesis: the bottleneck isn't retrieval breadth — it's that the hybrid pipeline already returns good candidates, and orchestrating more qkb commands adds cost without unlocking new information. The skill is now better for context efficiency (fewer Read calls), not for fact coverage.

Action items

  • Push graph as the default for relational questions. v0.2's clean traversal pattern (seed → Cypher path-match → walk) should be the model for graph-skill descriptions in other vaults too.
  • qkb's multi-get is underused. The bench showed agents reach for Read by reflex. Future skill iterations should explicitly direct agents to multi-get for 3+ file batches — but this is a behavioral nudge, not a capability gap.
  • Fix the Golden Brief Template wikilink in the graph workflow. Q10 agent surfaced a non-existent Golden Brief Capture Template — one of the wikilinks in Briefing Assembler.md may have drifted; vault hygiene needs a check.
  • Document the doc:N integer ID convention prominently in qkb's own docs (not just the skill). The Q7 bench agent thrashed through 50+ commands learning the schema; this should be in qkb's user-facing docs.
  • Skill triage hints should live in the description frontmatter. The "when to use" guidance above isn't visible to autonomous skill selection — moving it into each skill's description: field would improve auto-routing.

Methodology improvements for future runs

  • Run skill agents serially, not in parallel, when comparing skills that use qkb's reranker. Q9's vault-query (170s) and qkb (113s) inflated wallclock due to reranker contention; in isolation they'd be ~30-60s each.
  • Pick canonical pages before dispatching run-agents, not after. The eval agent needs to read those pages; locking them in before the bench eliminates a degree of evaluator subjectivity.
  • Verify wikilink targets in the ground-truth phase, not just citation-resolve-rate after. Q10 surfaced Golden Brief Capture Template as a dead link — could have caught this earlier.
