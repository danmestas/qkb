Caveman Compression Bench — flight-planner-kb hot.md

2026-05-15 · 4 conditions × 2 questions = 8 router runs · prose baseline vs caveman-compressed hot.md (standard / high / ultra levels)

Hypothesis

Caveman-compress the wiki/hot.md priming context fed to the vault-query router skill, then have the synthesis layer expand on the answer. Hypothesis: input tokens drop ~30–40% with answer fidelity holding within ±0.5 on the 0–5 grading scales.

Three compression levels tested with explicit fidelity rules preserving directional/causal connectives, wikilinks, numbers, code identifiers, and proper nouns:

First finding before the bench even ran. The standard agent compressed only 4.3% because flight-planner-kb's hot.md is already a telegraph-cadence operations log — wikilinks, file:line citations, PR/Linear references, code identifiers, dates. ~70% of the content is on the preserve-verbatim list. Articles, hedges, and conversational filler are sparse. The "ultra" agent compressed less than "high" because it took the fidelity rule more seriously. The premise that hot.md has lots of prose to cut was wrong for this vault.

Findings

  1. The hypothesis failed cleanly. Caveman compression on this vault's hot.md did NOT save tokens. Cave-ultra at 55% of original size saved 4.9% on tokens — well below the 30–40% reduction the hypothesis predicted. Cave-std saved 2.7% which is within noise. Cave-high increased token spend by 31%.
  2. Counterintuitive: more compression → more tokens. Cave-high (37 KB hot file, smallest of all four) used 31% more tokens than prose (114 KB hot file, largest). The pattern is consistent across both questions: FP-1 cave-high was 30% over prose, FP-2 cave-high was 32% over prose. Mechanism: the very-compressed hot file gave the agent less context to ride on, so it dispatched extra retrieval (qkb search + an extra page read) — and that extra retrieval's tokens dwarfed whatever was saved by reading a smaller priming file.
  3. The compression-cost curve is non-monotonic. Reading prose → std → ultra → high in increasing compression order, total tokens went 120k → 117k → 115k → 158k. There's a sweet spot around 50% of original size (cave-ultra @ 55%) where the file is smaller but the agent still trusts it. Past that knee, compression costs more than it saves.
  4. Answer quality held at all compression levels. All four conditions scored 15/15 on FP-1; prose scored 14/15 on FP-2 (missed 3 patterns the cave conditions surfaced), all three cave conditions scored 15/15. No quality degradation from compression on these questions. The fidelity rules in the prompt templates appear to have done their job — directional connectives, wikilinks, code identifiers all preserved.
  5. v5's FP-1 hallucination trap did not recur. Prose baseline correctly identified Briefing Assembler as the Go package. This is unrelated to caveman; it's a general "the prose baseline got better since v5" effect. The retrieval system is working better on this question than it was two days ago. Worth a separate investigation; not load-bearing for the caveman question.
  6. The premise about hot.md was wrong for this vault. flight-planner-kb's hot.md is an operational log, not prose. ~70% of its bytes are wikilinks, file:line citations, PR/Linear references, code identifiers, dates, and exact numbers — all on the preserve-verbatim list. There is no fat to cut at the "drop articles and filler" level. A vault with more narrative prose in hot.md might respond differently.

Implications

  • Don't ship caveman-compress for this vault. It saves no tokens at safe compression levels and costs tokens at aggressive levels. The intervention is wrong for the artifact.
  • Investigate the inverse pattern. Cave-high's extra retrieval was the synthesis layer compensating for sparseness. That's actually an information-preservation guarantee from the agent — it didn't hallucinate, it went and looked. Useful as a property; just expensive as a tactic.
  • Index.md is the better target. hot.md is already lean. wiki/index.md at 96 KB (~24k tokens) of "Page Name: one-line description" lookup rows is more structurally compressible — the schema is repetitive and the contract is "scan to find pointer, follow pointer." A different bench could test BM25-chunked index.md against full-read index.md, where the read-everything tax is much more obviously wasteful.
  • The compression target curve matters. If a future caveman pass aims at ~50% of original (cave-ultra range), it preserves trust and saves a small amount. Past that threshold, the agent stops trusting the file and the savings invert. The fidelity rule built into the ultra prompt template ("better sparse than hallucinated") was the right instinct — the agent that obeyed it produced the best result.
  • v5's hallucination floor is shifting. The fact that all four conditions correctly answered FP-1 — when v5's three skills all failed it — deserves a focused look. Either the vault content matured, the skill prompt got sharper, or it's a run-to-run draw. Re-running v5 with current skills would tell us which.

Methodology

8 retrieval runs dispatched as parallel general-purpose subagents, identical harness to v5. Each subagent received only the skill SKILL.md (or its compressed variant), the vault root, the collection name, and the question. Subagents were instructed to operate read-only — no qkb update, no file modifications, no answer-filing back to the vault. Token usage and elapsed time extracted from each subagent's <usage> block.

The three caveman compressions were each produced by a separate dispatched general-purpose subagent reading flight-planner-kb/wiki/hot.md and applying a prompt template from qkb/bench/caveman-prompts/{standard,high,ultra}.md. The prompt templates encode explicit preserve-verbatim rules for directional connectives, causal connectives, wikilinks, code identifiers, numbers, dates, and proper nouns. Fidelity rules ("better sparse than hallucinated") took precedence over compression targets, which is why cave-ultra ended up less compressed than cave-high.

Quality scoring used the same v5 rubric: coverage (0–5 hits the key facts), accuracy (0–5 citations resolve, claims correct), synthesis (0–5 actually answers vs lists). Scoring was hand-graded by the bench operator comparing each answer against the v5 graded baselines and the underlying vault pages (wiki/concepts/Briefing Assembler.md, wiki/concepts/Engine vs Assembler Boundary.md, plus the FP-2 pattern concept pages).

Reversibility: nothing in flight-planner-kb was modified. Compressed hot files lived under /tmp/qkb-cave-bench/ alongside the skill variants. Cleanup is a single rm -rf /tmp/qkb-cave-bench/. The bench artifacts (this report + the three prompt templates) live in the qkb repo under bench/results/ and bench/caveman-prompts/ respectively, on a feature branch.
