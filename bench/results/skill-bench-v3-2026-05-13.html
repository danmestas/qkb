The token finding

v1/v2 didn't surface this: qkb v0.2's multi-get orchestration actually saves tokens. Despite running more qkb commands than the other skills, it consumes the fewest cloud-LLM tokens (52.7k avg) because multi-get returns batched, line-budgeted content in one call — cheaper than 3–5 separate Read invocations.

vault-query is also cheap (52.9k) because grep-slicing meta-files keeps content out of context. bm25 reads more files (often 4–5) for context, accumulating tokens. graph burns tokens on Cypher exploration loops to find seed nodes.

So the cost picture is more nuanced than v1/v2 suggested:

  • Latency winners (vault-query, bm25): cheap wallclock when the question fits their model
  • Token winners (qkb v0.2, vault-query): cheap API cost via batched/grep'd content access
  • Coverage winner (graph): best quality on relational shape, mid on cost

The three axes (coverage, latency, tokens) don't fully correlate. A reasonable production routing strategy might be: bm25 for proper-noun lookups (fast + cheap-ish), qkb v0.2 for synthesis (cheapest tokens, mid latency), graph for relational (best coverage, worst latency + tokens), vault-query as the fallback when meta-files are small.

Ground truth (15 facts)
  1. Cycle-Aware Publication Refresh governs FAA sources publishing on calendar cycles (28-day AIRAC, 56-day, real-time).
  2. Pattern rule: download new data only when the cycle has advanced, not on fixed cadence.
  3. Goals: prevent wasted bandwidth, prevent mid-cycle corruption, make data version diagnosable.
  4. Sources governed: FAA NASR (28d), d-TPP (28d), Chart Supplements (56d), FAA DOF (56d+daily patches), ArcGIS airspace (56d).
  5. AIRAC is ICAO-standard 28-day cycle; FAA aligns with it.
  6. Implementation: compute current cycle, persist last_ingested_cycle per source in FaaPublicationCycle table.
  7. Reference: PreflightAPI's FaaPublicationCycleService.cs, FaaPublicationDateUtils.cs.
  8. AIRAC Delta Polling: two-phase FAA NMS NOTAM ingestion in Firestorm Dataworks.
  9. Phase 1 IL: full corpus on each new AIRAC via /v1/notams/il, AIXM 5.1 + SOAP, signed GCP URL.
  10. Phase 2 Delta: /v1/notams filtered by lastUpdatedDate (max 24h), GeoJSON preferred, sub-hourly.
  11. Scale 2026-05-02: 74,672 NOTAMs world across INT/DOM/MIL/FDC.
  12. Relation: Delta Polling = Cycle-Aware Refresh + delta-poll layer (NASR cycle-only, NMS IL + deltas).
  13. AIRAC cycles fixed decades in advance, internationally synchronized.
  14. Blocked endpoints (Staging 2026-05-02): /v1/notams/detail & /search return Apigee 404.
  15. Both patterns feed Lineage Invariant and Two-Tier Storage Architecture downstream.
Verdict: graph > vault-query = qkb > bm25. graph wins on coverage (87%) but pays in latency (65s) and tokens (54k). vault-query and qkb are tied for cheapest tokens (~48k); vault-query is fastest (25s) when it can grep meta-files. bm25 lost ground here because Cycle-Aware Publication Refresh + AIRAC Delta Polling are well-linked concept pages that benefit from graph topology.

Ground truth (18 facts)

Atomic Pipeline Handler (APH):

  1. One DagNats handler does archive + ingest as one atomic unit.
  2. Crystallized via AWC build Ousterhout review (split → merged).
  3. R2 keeps raw blob even if ingest fails; next cycle retries from scratch, no compensation logic.
  4. One handler = one retry surface, one observability budget.
  5. Pre-Ousterhout AWC split: archive + ingest as 2 tasks with depends_on — added orphan-archive failure mode.
  6. Split doubled handler count without separating concerns.

Schema-Drift Manifest (SDM):

  1. Defends against FAA/NOAA shipping schema changes without notice.
  2. Shared source_schemas table (migration 0005), reused across all pipelines.
  3. One table, one CLI, one runbook — beats per-pipeline *_parse_errors tables.
  4. Fingerprint = sha256 over sorted top-level JSON key set.
  5. Upsert with (source, file, fingerprint, blessed).
  6. Ingest worker proceeds regardless of blessed state; emits SchemaBlessed for observability.
  7. One table beats embedded JSON manifests (PreflightAPI Implementation A).

Cycle-Aware Publication Refresh (CAPR):

  1. FAA publishes on 28-day AIRAC cycles (NASR, d-TPP) or 56-day (DOF, charts, airspace) — schedule on cycles, not crons.
  2. Persist last_ingested_cycle in FaaPublicationCycle; pull only when current cycle advances.
  3. Prevents bandwidth waste (no 500 MB NASR re-pulls), prevents mid-cycle corruption.
  4. Alert when current cycle leads last_ingested by >24h.

Lineage Invariant + Two-Tier Storage Architecture:

  1. Every operational row carries source_object_key TEXT NOT NULL (Impl B — converged from Impl A's source_fetch_id); R2 archive immutable + Object-Locked; parsed tier derived/derivable.
Verdict: vault-query > qkb > graph > bm25. vault-query's breadth advantage wins here — only skill to cover all 5 canonical patterns (APH, SDM, CAPR, Lineage, Two-Tier) without omitting one. graph and bm25 both missed Cycle-Aware Publication Refresh entirely, a structural error for an FAA-ingestion question (FAA = cycles).

v3 findings

Token capture validates the protocol upgrade

The single new measurement (tokens per agent) flipped the cost story. v1/v2 framed the comparison as latency vs coverage; v3 reveals tokens as a third orthogonal axis where the rankings differ:

  • Latency rank: vault-query > bm25 > graph > qkb
  • Coverage rank: graph ≈ vault-query > qkb > bm25
  • Token rank: qkb ≈ vault-query > graph ≈ bm25

No single skill wins all three axes on this question set. The right routing strategy depends on which cost matters most.

qkb's multi-get is finally measurable

In v2 the qkb skill's full-toolkit orchestration looked like a wash on quality. v3 shows it's a real token efficiency win — qkb v0.2 consumes 52.7k avg tokens, fewest of any skill, by using qkb multi-get to batch-fetch trimmed file content instead of 3–5 separate Read calls. The latency tradeoff (62s avg, slowest) buys the lowest API cost.

vault-query has a context cliff

Q12 (small concept pages): vault-query was fastest AND cheapest. Q13 (multi-pattern synthesis spread across 5 concept pages): vault-query still won on coverage by reading 4 distinct pages directly. But the v2 Q4/Q5 runs showed it degrading to 20-64% when meta-files exceeded read budget. The recommendation stands: vault-query is the fast/cheap choice conditional on the vault's hot/index pages fitting context.
